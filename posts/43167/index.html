一、Git概述

  Git 是一个免费的、开源的分布式版本控制系统，可以快速高效地处理从小型到大型的各种 项目。

  Git 易于学习，占地面积小，性能极快。 它具有廉价的本地库，方便的暂存区域和多个工作 流分支等特性。其性能优于 Subversion、CVS、Perforce 和 ClearCase 等版本控制工具。

【官网】


1、什么是版本控制

  版本控制是一种记录文件内容变化，以便将来查阅特定版本修订情况的系统。

  版本控制其实最重要的是可以记录文件修改历史记录，从而让用户能够查看历史版本， 方便版本切换。




2、为什么需要版本控制

个人开发过渡到团队协作。




3、版本控制工具

  • 集中式版本控制工具

   CVS、SVN(Subversion) 、VSS……

  集中化的版本控制系统诸如 CVS、SVN 等，都有一个单一的集中管理的服务器，保存 所有文件的修订版本，而协同工作的人们都通过客户端连到这台服务器，取出最新的文件或 者提交更新。多年以来，这已成为版本控制系统的标准做法。

  这种做法带来了许多好处，每个人都可以在一定程度上看到项目中的其他人正在做些什 么。而管理员也可以轻松掌控每个开发者的权限，并且管理一个集中化的版本控制系统，要 远比在各个客户端上维护本地数据库来得轻松容易。

  事分两面，有好有坏。这么做显而易见的缺点是中央服务器的单点故障。如果服务器宕 机一小时，那么在这一小时内，谁都无法提交更新，也就无法协同工作。

  • 分布式版本控制工具

  Git、Mercurial、Bazaar、Darcs……

  像 Git 这种分布式版本控制工具，客户端提取的不是最新版本的文件快照，而是把代码 仓库完整地镜像下来（本地库）。这样任何一处协同工作用的文件发生故障，事后都可以用 其他客户端的本地仓库进行恢复。因为每个客户端的每一次文件提取操作，实际上都是一次 对整个文件仓库的完整备份。

  分布式的版本控制系统出现之后，解决了集中式版本控制系统的缺陷:

    服务器断网的情况下也可以进行开发（因为版本控制是在本地进行的）

    每个客户端保存的也都是整个完整的项目（包含历史记录，更加安全）

1、什么是分支

  在版本控制过程中，同时推进多个任务，为每个任务，我们就可以创建每个任务的单独 分支。使用分支意味着程序员可以把自己的工作从开发主线上分离开来，开发自己分支的时 候，不会影响主线分支的运行。对于初学者而言，分支可以简单理解为副本，一个分支就是 一个单独的副本。（分支底层其实也是指针的引用）

分支的好处：

  同时并行推进多个功能开发，提高开发效率。各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响。失败的分支删除重新开始即可。

步骤 1：查看远程分支列表

首先，你可以查看远程分支的列表，以确定远程分支的名称。

这将显示所有远程分支。

git fetch

git fetch：更新远程分支的引用

  • 作用：
    • git fetch 命令从远程仓库获取最新的变更，这些变更包括新的提交、新的分支、更新的引用等，但不会自动合并到你的本地分支中。
    • 当你执行 git fetch 时，Git 会从远程仓库中拉取所有更新，但不会改变你的本地代码，只会更新远程引用。
  • 为什么需要 git fetch：
    • 在执行 git branch -r 之前，通常建议运行 git fetch，这样可以确保你看到的是最新的远程分支信息。
    • 如果不先运行 git fetch，那么 git branch -r 列出的远程分支列表可能是陈旧的，因为远程仓库中的分支有可能发生了变化（例如新增、删除或更新），而你的本地 Git 未获取到最新的状态。

步骤 2：创建一个本地分支来追踪远程分支

命令解释

  • git checkout -b feature-branch origin/feature-branch：
    • git checkout -b：创建并切换到一个新的本地分支。
    • feature-branch：本地新分支的名字，你可以选择任意名字。
    • origin/feature-branch：远程分支的名字，表示从远程的 origin 拉取分支 feature-branch 的内容。

通过这条命令，你会创建一个名为 feature-branch 的本地分支，并且基于远程的 feature-branch 分支内容。它不会将远程的更改直接合并到你当前的分支。

7.4.2 创建tag

在 Git 中，有两种类型的标签：轻量级标签（lightweight）和注释标签（annotated）。这两种标签在用途和存储方式上有所不同。


轻量级标签

轻量级标签（lightweight tag）基本上是一个指向特定提交的引用，就像一个不可变的分支。它简单地记录了某个提交的哈希值。

特点：

  • 不包含附加的元数据：轻量级标签不存储任何除了指向的提交之外的数据，如创建标签的用户、时间等信息。
  • 创建快速简单：由于不需要额外的信息，创建轻量级标签非常快速且简单。
  • 用途：通常用于临时标记某个提交以便将来引用，或者在不需要标签的附加信息的场景中使用。

注释标签

相对于轻量级标签，注释标签（annotated tag）则包含更多的信息。

特点：

  • 包含作者信息：注释标签存储创建者的信息，包括名字和电子邮件地址。
  • 包含消息：注释标签允许附带一条消息，通常用于记录标签的发布说明或其他重要信息。
  • 包含创建日期：注释标签记录了标签被创建的日期。
  • 使用 GPG 签名：注释标签可以被签名和验证，这是确保标签从未被篡改的一个重要安全特性。

总结

轻量级标签和注释标签的选择取决于你对标签附加信息的需求。轻量级标签更简单，适合快速标记；而注释标签则更适合正式的发布，因为它们提供了更多关于标签历史和目的的信息。在版本发布时推荐使用注释标签，因为它们的附加信息可以帮助维护项目的历史记录和透明度。

7.7 git ignore

在使用 Git 进行版本控制时，.gitignore 文件扮演着至关重要的角色。它用于指定哪些文件或目录应被 Git 忽略，不纳入版本控制。正确配置 .gitignore 可以帮助保持代码库的整洁，避免不必要的文件被提交，提高团队协作效率。


  1. 什么是 .gitignore

.gitignore 是一个文本文件，位于 Git 仓库的根目录或特定子目录中，包含一系列规则，用于告诉 Git 忽略哪些文件和目录。这些规则可以基于文件名、目录名、文件扩展名或模式匹配。


  1. .gitignore 文件的位置

根目录：

  • 推荐位置：通常将 .gitignore 文件放在 Git 仓库的根目录下。这确保了整个项目中的所有子目录和文件都受到忽略规则的影响。
  • 优点：
    • 集中管理忽略规则，简化维护。
    • 易于新成员理解和遵循。

子目录：

  • 可选位置：在特定子目录中也可以创建 .gitignore 文件，以覆盖或添加更具体的忽略规则。
  • 使用场景：
    • 某些子目录有特殊的忽略需求，例如生成大量临时文件的目录。

模式匹配：

  • 通配符 *：匹配零个或多个字符。

    • *.log 忽略所有 .log 文件。
    • temp* 忽略所有以 temp 开头的文件或目录。

  • 斜杠 /：用于指定目录。

    • logs/ 忽略 logs 目录及其所有内容。
    • /config/*.json 只忽略根目录下的 config 目录中的 .json 文件。

  • 感叹号 !：用于排除某些被忽略的文件。

    • *.log 忽略所有 .log 文件。
    • !important.log 但不忽略 important.log 文件。

  • 双星号 **：匹配任意层级的目录。

    • **/logs 忽略所有层级中的 logs 目录。

7.8 git stash

Git Stash 是 Git 版本控制系统中一个非常实用的特性，允许你临时保存（“stash”）你的工作目录中的修改，而不必提交这些改动。这对于当你需要快速切换分支处理其他任务，但你的当前分支又有未完成的工作时非常有用。


Stash 当前工作

  1. Stashing 改动:

    • 打开终端（命令行）。
    • 切换到你的项目目录下。
    • 执行 git stash 或 git stash push 命令来保存当前工作目录和暂存区的改动。
    • 你可以给 stash 加上一个描述，使之后容易识别：git stash push -m "Your custom message here"。

    这会将当前的修改保存到一个临时区域（Stash），工作目录会回到最近的 commit 状态。

  2. 查看 Stash 列表:

    • 使用 git stash list 查看当前保存的 stash 列表。

从 Stash 中取出改动

  1. 应用 Stash:
    • 执行 git stash apply 来应用最近一次的 stash 到当前工作目录。如果你有多个 stash，你可以指定一个特定的 stash 来应用，如：git stash apply stash@{0}。
    • 使用 git stash pop 将改动重新应用到工作目录并从 stash 列表中移除这条记录。例如：git stash pop stash@{0}。

清除 Stash

  • 使用 git stash clear 来删除所有缓存的 stash。

应用场景

  1. 多任务处理:

    • 当你正在一个分支上开发新功能，但突然需要切换到另一个分支处理紧急修复。你可以使用 git stash 保存当前进度，处理完紧急任务后，再用 git stash pop 恢复工作。

  2. 合并冲突:

    • 在合并或重置操作前，你可能不想提交半成品的代码。使用 git stash 保存当前工作状态，然后安全地执行合并或重置。操作完成后，再通过 git stash pop 恢复你的工作。

  3. 试验性改动:

    • 当你想尝试某些代码改动但不确定是否要保留时，可以先 stash 现有工作，之后随时可以恢复或废弃。

注意事项

  • Stash 是本地的，不会随着代码库一起推送到远程仓库。
  • 使用 git stash apply 或 git stash pop 可能会遇到冲突，你需要手动解决这些冲突。
  • Stash 可能不会保存未跟踪的文件（如全新添加的文件），除非你使用 git stash -u 或 git stash --include-untracked。

通过合理使用 Git Stash，你可以更灵活地管理工作流，处理多任务而不会丢失任何进展。

7.10 场景案例

场景1：同事A的分支feat/branch_a正在测试环境测试，小B想部署自己的分支feat/branch_b进行测试

实践做法：

小B git checkout -b feat/branch_b_test，再 git rebase feat/branch_a ，然后即可直接部署 feat/branch_b_test ，不会影响到同事A的测试。

如果小B想要修改代码并继续测试，可以更新 feat/branch_b，然后再回到 feat/branch_b_test， 并 git rebase feat/branch_b，然后接着部署即可。


场景2：同事A和同事B共同完成一项开发任务

实践做法：

构建一条主干分支 feat/task, 小A和小B在这个分支上checkout出新的分支feat/task_a和feat/task_b，然后要发测试环境测试的时候，再合入主干分支，让对方再重新rebase主干分支
